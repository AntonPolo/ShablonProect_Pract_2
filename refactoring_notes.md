# Краткая записка по применению Фабричного метода

## Принятые решения
1. Сохранена структура из Pract1: `Request`, `RequestSource`, `RequestService`.
2. Реализованы три учебные реализации источника данных: база, файл, веб.
3. Фабрика `RequestSourceFactory` читает `config.properties` и создаёт нужную реализацию.
4. `RequestService` получает `RequestSource` через конструктор — не зависит от конфигурации.

## Почему Фабричный метод?
Паттерн применён, потому что:
- Тип источника определяется внешней конфигурацией,
- Выбор реализации централизован,
- Доменная логика работает только с абстракцией.

## Улучшение тестируемости
Теперь в тестах можно:
- Передавать любую реализацию `RequestSource`,
- Полностью изолировать бизнес-логику от инфраструктуры,
- Моделировать сценарии "дубль" / "новый запрос" без внешних зависимостей.
